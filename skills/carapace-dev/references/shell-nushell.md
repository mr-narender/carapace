# Carapace Library: Nushell Shell Integration Deep Dive

Reference for [carapace](https://github.com/carapace-sh/carapace)'s nushell completion integration — how the snippet works, how completion output is formatted, and how carapace handles nushell-specific edge cases including open-quote patching, JSON output, style conversion, and nospace via trailing space. For cross-shell comparisons, see the **references/shell.md**.

## Source Files

| File | Purpose |
|------|---------|
| `internal/shell/nushell/snippet.go` | Nushell completion closure generation |
| `internal/shell/nushell/action.go` | Value formatting, JSON serialization, quoting/nospace |
| `internal/shell/nushell/patch.go` | Open-quote stripping for args starting with `"`, `'`, or `` ` `` |
| `internal/shell/nushell/style.go` | Style conversion from carapace format to nushell `{fg, bg, attr}` |
| `internal/shell/shell.go` | Shared dispatch — message integration, nospace propagation, filtering |
| `complete.go` | Entry point — calls `nushell.Patch()` for shell-specific arg patching |

## Nushell Completion System Background

### External Completers

Nushell registers external completers via a closure in `$env.config.completions.external`:

```nu
let carapace_completer = {|spans|
    carapace $spans.0 nushell ...$spans | from json
}

$env.config.completions.external = {
    enable: true
    max_results: 100
    completer: $carapace_completer
}
```

The completer closure receives `spans` — a `list<string>` of command-line arguments parsed by nushell:

- `spans.0` is the command name (e.g., `"git"`)
- Remaining elements are the arguments already typed
- A trailing empty string or space represents the cursor position

When the closure returns `null`, nushell falls back to built-in file completion.

### The `from json` Pipeline

Carapace's completer outputs JSON to stdout, which is piped through nushell's `from json` command to produce nushell records. This is fundamentally different from shells like bash/zsh/fish that parse raw text output — nushell receives **structured data** with typed fields.

### Nushell Completion Record Fields

Nushell's completion system (via Reedline's `Suggestion` struct) accepts records with these fields:

| Field | Type | Purpose |
|-------|------|---------|
| `value` | string | Text inserted into the command line |
| `display` | string | Text shown in the completion menu (overrides `value`) |
| `description` | string | Help text shown alongside the suggestion |
| `style` | string or record | Styling — color name, hex code, or `{fg, bg, attr}` record |
| `span` | record | `{start: int, end: int}` byte range to replace |
| `append_whitespace` | bool | Whether to auto-append trailing space after insertion |

Carapace uses a subset of these fields: `value`, `display`, `description`, and `style`. It does **not** use `span` (letting nushell compute it) or `append_whitespace` (encoding nospace directly in the `value` field via trailing space).

### Nushell String Types and Quoting

Nushell has several string types that affect how completion values are inserted:

| Type | Syntax | Escapes | Use in completions |
|------|--------|---------|-------------------|
| Single-quoted | `'text'` | None | Values with special chars |
| Double-quoted | `"text"` | C-style (`\"`, `\\`, etc.) | Values needing escape sequences |
| Raw string | `r#'text'#` | None | Values containing single quotes |
| Bare word | `word` | None | Simple alphanumeric values only |
| Backtick | `` `text` `` | None | Paths with spaces |

Characters that require quoting in nushell: ` ` (space), `{`, `}`, `(`, `)`, `[`, `]`, `$`, `"`, `'`, `` ` ``, `<`, `>`, `&`, `|`, `;`, `#`, `\`.

## The Nushell Snippet

Generated by `nushell.Snippet(cmd)` in `internal/shell/nushell/snippet.go`:

```nu
let example_completer = {|spans| 
    example _carapace nushell ...$spans | from json
}
```

### Snippet Walkthrough

**1. Closure definition**

```nu
let example_completer = {|spans|
```

The `let` binding creates a closure variable. Nushell closures are first-class values that can be stored and passed around. The `|spans|` parameter receives the command-line spans as a `list<string>`.

**2. Command invocation**

```nu
    example _carapace nushell ...$spans | from json
```

- `example` — the CLI being completed (the command name from `cmd.Name()`)
- `_carapace` — carapace's hidden completion subcommand
- `nushell` — the shell name, telling carapace to use the nushell formatter
- `...$spans` — spread operator: passes each span as a separate argument to `_carapace`
- `| from json` — nushell pipeline: parses the JSON output into nushell records

The `...$spans` spread is critical — it passes `[example, subcommand, --flag, value]` as separate arguments, exactly as carapace expects.

**3. Registration**

The closure must be registered in nushell's config. The typical pattern is:

```nu
# In env.nu or config.nu
let carapace_completer = {|spans|
    CARAPACE_LENIENT=1 carapace $spans.0 nushell ...$spans | from json
}

$env.config.completions.external = {
    enable: true
    max_results: 100
    completer: $carapace_completer
}
```

The `CARAPACE_LENIENT=1` environment variable is commonly set for nushell to prevent carapace from returning errors for unknown flags during completion (since nushell may pass partial or incomplete flags).

### Why the Snippet Is So Simple

The nushell snippet is just two lines because:

- Nushell's parser handles quoting natively and passes clean spans
- Nushell's `...$spans` spread operator handles argument passing without `xargs`
- Nushell doesn't need special environment variables like `COMP_LINE` or `CARAPACE_COMPLINE`
- The closure is native nushell code (no shell script wrapping)

However, the simplicity of the snippet belies the complexity handled on the Go side by `nushell.Patch()` and the JSON formatter.

## The Patch Phase: `nushell.Patch()`

Before traversal, `complete()` in `complete.go` calls `nushell.Patch(args)` when the shell is nushell:

```go
case "nushell":
    args = nushell.Patch(args) // handle open quotes
```

This is the only shell besides bash and cmd-clink that requires Go-side argument patching.

### Implementation

```go
func Patch(args []string) []string {
    for index, arg := range args {
        if len(arg) == 0 {
            continue
        }

        switch arg[0] {
        case '"', "'"[0]:
            if tokens, err := shlex.Split(arg); err == nil {
                args[index] = tokens[0].Value
            }
        case '`':
            args[index] = strings.Trim(arg, "`")
        }
    }
    return args
}
```

### What It Does

Nushell's parser passes arguments to the completer closure with **quoting intact**. When a user is completing inside quotes, nushell includes the quote characters in the span:

| User types | Span received by completer | After Patch |
|-----------|---------------------------|-------------|
| `example action` | `"action"` | `"action"` |
| `example "act` | `"\"act"` or `'"act'` | `"act"` |
| `example 'act` | `"'act"` or `"'act"` | `"act"` |
| `example `` `act` `` | `` "`act"` `` | `"act"` |

For each argument:

1. **Empty args** — skipped (common for cursor-at-end positions)
2. **Double-quoted args** (`"..."`) — parsed via `shlex.Split` to extract the inner value
3. **Single-quoted args** (`'...'`) — parsed via `shlex.Split` to extract the inner value
4. **Backtick-quoted args** (`` `...` ``) — trimmed of backticks directly via `strings.Trim`

### Why shlex.Split for Quotes

The `shlex.Split` function (from [carapace-shlex](https://github.com/carapace-sh/carapace-shlex)) properly handles:

- Fully quoted strings: `"hello"` → `hello`
- Open-quoted strings: `"hello` → `hello` (shlex closes the quote)
- Escaped characters inside double quotes: `"hello\"world"` → `hello"world`
- Single-quoted strings: `'hello'` → `hello`

Backtick-quoted strings are handled separately with `strings.Trim` because nushell's backtick syntax is not POSIX shell quoting — it's a nushell-specific bare string that can contain spaces but not escapes.

### Known Limitations

1. **Mixed quoting** — If a span contains mixed quote types (e.g., `"it's"`), `shlex.Split` handles the outer quotes correctly but the inner quote is preserved as-is.

2. **Partial quoting** — An open quote like `"partial` is handled by shlex (which closes the quote internally), but the closing quote character is not present in the completion output. The nushell formatter's quoting logic (in `ActionRawValues`) adds quotes back if needed.

3. **No redirect handling** — Unlike bash, nushell does not leak redirect operators into the completion arguments, so no redirect filtering is needed.

4. **No COMP_WORDBREAKS equivalent** — Nushell's parser doesn't split on `:` or `=`, so no wordbreak prefix handling is needed.

## Value Formatting: `ActionRawValues()`

`nushell.ActionRawValues(currentWord, meta, values)` in `internal/shell/nushell/action.go` formats completion candidates as a JSON array of records. It is one of the richer shell formatters in carapace — handling quoting, nospace, style conversion, and JSON serialization.

### Output Format

```json
[
  {"value": "branch ", "display": "branch", "description": "Switch branches", "style": {"fg": "green"}},
  {"value": "checkout ", "display": "checkout", "description": "Switch working tree"},
  {"value": "main", "display": "main", "description": "Default branch", "style": {"fg": "red"}}
]
```

Each record has:

| Field | Always present | Source |
|-------|---------------|--------|
| `value` | Yes | `RawValue.Value` — with quoting applied and trailing space for nospace |
| `display` | Yes | `RawValue.Display` — the human-readable display text |
| `description` | Only when non-empty | `RawValue.TrimmedDescription()` |
| `style` | Only when non-empty | `RawValue.Style` — converted to nushell format via `convertStyle()` |

The `omitempty` JSON tags mean `description` and `style` are omitted when empty, keeping the output compact.

### The `record` Struct

```go
type record struct {
    Value       string        `json:"value"`
    Display     string        `json:"display"`
    Description string        `json:"description,omitempty"`
    Style       *nushellStyle `json:"style,omitempty"`
}
```

### Step 1: Sanitize

```go
var sanitizer = strings.NewReplacer(
    "\n", ``,
    "\r", ``,
)

func sanitize(values []common.RawValue) []common.RawValue {
    for index, v := range values {
        (&values[index]).Value = sanitizer.Replace(v.Value)
        (&values[index]).Display = sanitizer.Replace(v.Display)
        (&values[index]).Description = sanitizer.Replace(v.Description)
    }
    return values
}
```

Newlines and carriage returns are stripped from all fields. Unlike the fish formatter, **tabs are NOT stripped** because nushell's JSON format does not use tabs as field delimiters — values are structured in JSON, not tab-separated.

### Step 2: Quote Values with Special Characters

```go
if strings.ContainsAny(val.Value, ` {}()[]<>$&"'|;#\`+"`") {
    switch {
    case strings.HasPrefix(val.Value, "~"):
        val.Value = fmt.Sprintf(`~"%v"`, escaper.Replace(val.Value[1:]))
    default:
        val.Value = fmt.Sprintf(`"%v"`, escaper.Replace(val.Value))
    }
}
```

Values containing nushell metacharacters are double-quoted. The check uses `strings.ContainsAny` against:

```
{ } ( ) [ ] < > $ & " ' | ; # \ ` (backtick) space
```

Two quoting modes:

**Tilde-prefixed values** (`~"rest"`): Tilde is kept outside the double quotes to preserve nushell's tilde expansion. The rest of the path is double-quoted. Example: `~/Documents/My File.txt` → `~"/Documents/My File.txt"`.

**All other values** (`"value"`): The entire value is wrapped in double quotes. Example: `hello world` → `"hello world"`.

The `escaper` handles escaping inside double quotes:

```go
var escaper = strings.NewReplacer(
    `\`, `\\`,
    `"`, `\"`,
)
```

Only backslash and double-quote need escaping inside nushell double-quoted strings. This is simpler than bash/zsh which need to escape many more characters.

### Step 3: Nospace — Trailing Space Encoding

```go
nospace := meta.Nospace.Matches(val.Value)
if !nospace {
    val.Value = val.Value + " "
}
```

Carapace encodes the nospace signal as a **trailing space** in the `value` field:

- **`nospace = true`** — value ends with no trailing space (nushell won't add one either)
- **`nospace = false`** — value ends with a space character, so nushell inserts the value **with** the trailing space

This is a clever use of nushell's JSON format — since the `value` field is the exact text inserted into the command line, a trailing space in the value becomes a trailing space after insertion. Nushell's `append_whitespace` field defaults to `false`, so carapace controls spacing entirely through the `value` field.

This approach means:
- Values like `--flag=` (with `NoSpace('=')`) are inserted without trailing space
- Values like `checkout` (no nospace) get `"checkout "` — inserted with trailing space
- Values like `/path/to/dir/` (with `NoSpace('/')`) are inserted without trailing space

### Step 4: Build Records and Marshal JSON

```go
vals[index] = record{
    Value:       val.Value,
    Display:     val.Display,
    Description: val.TrimmedDescription(),
    Style:       convertStyle(val.Style),
}

m, _ := json.Marshal(vals)
return string(m)
```

Each value becomes a `record` with the processed `Value`, original `Display`, truncated `Description`, and converted `Style`. The entire array is marshaled to JSON.

## Style Conversion: `convertStyle()`

`convertStyle(_style string)` in `internal/shell/nushell/style.go` translates carapace's style string format into nushell's `{fg, bg, attr}` record format.

### The `nushellStyle` Struct

```go
type nushellStyle struct {
    Foreground string `json:"fg,omitempty"`
    Background string `json:"bg,omitempty"`
    Attributes string `json:"attr,omitempty"`
}
```

When empty style is passed, `convertStyle` returns `nil` (omitted from JSON via `omitempty`).

### Style Parsing Pipeline

1. `style.Parse(_style)` — parses carapace's style string (e.g., `"bold red"`, `"#ff0000"`, `"fg-red bg-blue bold"`) into a structured `style.Style` with `Foreground`, `Background`, `Bold`, `Italic`, etc. fields.

2. **Attribute mapping** — carapace attributes are mapped to nushell's single-letter attribute flags:

   | Carapace Attribute | Nushell Flag | Meaning |
   |-------------------|---------------|---------|
   | `Blink` | `l` | Blinking text |
   | `Bold` | `b` | Bold text |
   | `Dim` | `d` | Dim/faint text |
   | `Italic` | `i` | Italic text |
   | `Inverse` | `r` | Reverse video (swap fg/bg) |
   | `Underlined` | `u` | Underlined text |

   Multiple attributes are concatenated: e.g., bold+underline → `"bu"`.

3. **Color conversion** — `convertColor()` maps carapace colors to nushell color names:

   **Named colors** — Mapped to nushell equivalents:

   | Carapace Name | Nushell Name |
   |---------------|-------------|
   | `black` | `black` |
   | `red` | `red` |
   | `green` | `green` |
   | `yellow` | `yellow` |
   | `blue` | `blue` |
   | `magenta` | `magenta` |
   | `cyan` | `cyan` |
   | `white` | `white` |
   | `bright-black` | `dark_gray` |
   | `bright-red` | `light_red` |
   | `bright-green` | `light_green` |
   | `bright-yellow` | `light_yellow` |
   | `bright-blue` | `light_blue` |
   | `bright-magenta` | `light_magenta` |
   | `bright-cyan` | `light_cyan` |
   | `bright-white` | `white` |

   **XTerm256 colors** (`color0` through `color255`) — Converted to hex RGB strings (e.g., `color196` → `#ff0000`). Each XTerm256 color is mapped to its exact hex equivalent.

   **Hex colors** (`#RRGGBB`) — Passed through unchanged.

### JSON Output Examples

**Simple colored value:**
```json
{"value": "error ", "display": "error", "description": "something went wrong", "style": {"fg": "red"}}
```

**Bold underline with hex background:**
```json
{"value": "important ", "display": "important", "style": {"fg": "#ffffff", "bg": "#ff0000", "attr": "bu"}}
```

**No style:**
```json
{"value": "plain ", "display": "plain"}
```

### Known Limitations in Style Conversion

1. **`bright-white` and `white` collision** — Both map to `white` in nushell. Nushell doesn't have a separate "bright-white" named color; it would need `#ffffff` vs `#c0c0c0` hex to distinguish them. The code has TODO comments about this.

2. **`bright-black` → `dark_gray`** — Nushell uses `dark_gray` where many terminals call this `bright-black` or `gray`. The mapping is correct for nushell but may look different from other shells.

3. **Attribute string ordering** — Attributes are concatenated in a fixed order (`l`, `b`, `d`, `i`, `r`, `u`). Nushell doesn't mandate a specific order for the attribute string, but carapace produces a consistent ordering.

## Nospace Handling

### How Carapace Encodes Nospace

Unlike most other shells, nushell's JSON-based output format naturally supports **per-candidate nospace** — each record's `value` field is the exact text that gets inserted into the command line, including any trailing space.

The implementation in `ActionRawValues` is straightforward:

```go
nospace := meta.Nospace.Matches(val.Value)
if !nospace {
    val.Value = val.Value + " "
}
```

When `meta.Nospace.Matches(val.Value)` returns `true` (the value's last character is in the nospace set, or `*` is in the set), no trailing space is added. Otherwise, `" "` is appended.

### Common Nospace Patterns

| Pattern | Action | Nospace Effect |
|---------|--------|---------------|
| Directories ending in `/` | `ActionFiles().NoSpace('/')` | `"/path/to/dir/"` — no trailing space |
| Branch refs like `origin/main` | `ActionValues(...).NoSpace('/')` | `"origin/main"` — no trailing space |
| Comma-separated values | `ActionValues(...).NoSpace(',')` | `"first,second,"` — no trailing space |
| Flag values like `--flag=` | `ActionValues(...).NoSpace('=')` | `"--flag="` — no trailing space |
| Regular completions | (default) | `"checkout "` — trailing space added |

### Interaction with shell.go

In `shell.go`, before `ActionRawValues` is called, the shared dispatch may add `*` to the nospace set:

```go
if shell != "export" {
    switch {
    case !meta.Messages.IsEmpty():
        meta.Nospace.Add('*')    // messages ⇒ force nospace on ALL values
    case env.Nospace() != "":
        meta.Nospace.Add([]rune(env.Nospace())...)  // CARAPACE_NOSPACE env var
    }
}
```

When messages exist, `Nospace.Add('*')` forces **all** values to be nospace. This prevents nushell from auto-inserting error message values with a trailing space.

## Message Handling

Nushell falls into the **`default`** case in `shell.go`'s message handling:

```go
switch shell {
case "elvish", "export", "zsh": // shells with support for showing messages
default:
    values = meta.Messages.Integrate(values, value)
}
```

This means messages are **integrated as synthetic completion candidates** via `meta.Messages.Integrate(values, value)`. This method converts messages into fake completion values:

- Display text: `ERR`, `ERR1`, `ERR2`, etc. (incrementing to avoid collisions with real values)
- Value text: `<currentWord prefix>ERR`, `<currentWord prefix>ERR1`, etc.
- Description: the actual error message text
- Style: `style.Carapace.Error`

When messages exist, `meta.Nospace.Add('*')` is also set, preventing any trailing space on error completions.

## Edge Cases and Known Issues

### 1. Open-Quote Arguments

Nushell passes quoted arguments to the completer with quote characters intact. When the user is typing inside an unclosed quote:

```
example --flag "partial_wor⇥
```

Nushell may pass `'"partial_wor'` or `'"partial_wor'` as the span (with the opening quote included). The `nushell.Patch()` function handles this by:

1. Detecting arguments starting with `"` or `'`
2. Parsing them through `shlex.Split` to strip the quotes
3. Replacing the original span with the unquoted value

For backtick-quoted strings (`` `partial_wor` ``), `strings.Trim(arg, "`")` strips the backticks directly.

**Limitation**: Only fully-quoted or simply-open-quoted arguments are handled correctly. Complex nested quoting scenarios (e.g., `"it's a 'test"`) may not be stripped perfectly by `shlex.Split`.

### 2. Quoting of Completion Values

When a completion value contains nushell metacharacters, `ActionRawValues` wraps it in double quotes:

```go
if strings.ContainsAny(val.Value, ` {}()[]<>$&"'|;#\`+"`") {
    switch {
    case strings.HasPrefix(val.Value, "~"):
        val.Value = fmt.Sprintf(`~"%v"`, escaper.Replace(val.Value[1:]))
    default:
        val.Value = fmt.Sprintf(`"%v"`, escaper.Replace(val.Value))
    }
}
```

**Tilde values** — Paths starting with `~` are handled specially: the tilde is kept outside the double quotes so nushell can expand it. `~/My Documents/file.txt` → `~"/My Documents/file.txt"`.

**Other values** — All metacharacters trigger double-quoting: `hello world` → `"hello world"`, `file(1).txt` → `"file(1).txt"`.

**Backslash and double-quote escaping** — The `escaper` handles `\\` and `\"` inside double-quoted strings. Other characters don't need escaping inside nushell double quotes.

### 3. JSON Parsing Errors

The snippet uses `| from json` to parse carapace's output. If carapace returns invalid JSON (e.g., due to an internal error), `from json` will fail and produce no completions. The user sees an empty completion list.

Common causes:
- Carapace crashes during completion
- Unknown flags when `CARAPACE_LENIENT` is not set — carapace returns an error message instead of completions
- Output buffering issues (rare, but possible with large completion sets)

**Mitigation**: Setting `CARAPACE_LENIENT=1` in the completer closure prevents unknown-flag errors:

```nu
let carapace_completer = {|spans|
    CARAPACE_LENIENT=1 carapace $spans.0 nushell ...$spans | from json
}
```

### 4. The `from json` Pipeline and `null` Handling

When carapace returns an empty completion set (no matches), `ActionRawValues` produces `[]` (an empty JSON array). The `from json` command converts this to an empty nushell list, which the completion system treats as "no completions" and falls back to file completion.

However, if carapace returns `null` (e.g., from an error), `from json` may produce `null`, which triggers nushell's built-in file completion fallback. This is generally the desired behavior.

### 5. Alias Expansion Bug

Nushell has a known bug where autocompletions don't work for aliases. When completing an aliased command, `$spans.0` contains the alias name, not the expanded command. Carapace may not recognize the alias and return no completions.

**Workaround** (commonly used in carapace-bin setups):

```nu
let carapace_completer = {|spans|
    let expanded_alias = (scope aliases | where name == $spans.0 | get -i 0 | get -i expansion)
    let spans = (if $expanded_alias != null {
        $spans | skip 1 | prepend ($expanded_alias | split row " " | take 1)
    } else {
        $spans
    })
    CARAPACE_LENIENT=1 carapace $spans.0 nushell ...$spans | from json
}
```

This looks up the alias, expands it, and replaces `$spans.0` with the actual command name.

### 6. No Redirect Filtering Needed

Unlike bash, nushell does not leak redirect tokens (`>`, `<`, `>>`, `2>`) into the completion arguments. Nushell's parser handles redirects internally, so `nushell.Patch()` does not need to filter them.

### 7. No COMP_WORDBREAKS Equivalent

Nushell's tokenizer does not split on `:`, `=`, `@`, or other word-break characters. The spans passed to the completer already contain the full word. This means:

- `origin/main` stays as one span (not split at `/`)
- `--flag=value` stays as one span (not split at `=`)
- `user@host` stays as one span (not split at `@`)

This is a significant advantage over bash, which requires complex COMP_WORDBREAKS handling.

### 8. Description Truncation

Carapace uses `val.TrimmedDescription()` in the nushell formatter, which truncates descriptions exceeding `CARAPACE_DESCRIPTION_LENGTH` (default 80 characters) with an ellipsis. This keeps the JSON output compact and prevents nushell's completion menu from becoming unreadable with overly long descriptions.

### 9. Style Omission for Empty Styles

When a `RawValue` has no style (`val.Style == ""`), `convertStyle()` returns `nil`. The `omitempty` JSON tag on the `Style` field causes it to be omitted entirely from the output:

```json
{"value": "plain ", "display": "plain"}
```

This reduces JSON output size compared to including `"style": {}` or `"style": null`.

### 10. Nushell Process Name Detection

Shell detection in `pkg/ps/ps.go` identifies nushell by the process name `"nu"`:

```go
case "nu":
    return "nushell"
```

The nushell binary is named `nu`, not `nushell`. The mapping converts the process name to carapace's internal shell identifier `"nushell"`.

## Completion Dispatch Flow for Nushell

```
User presses TAB
  → Nushell invokes the completer closure
  → Snippet: carapace $spans.0 nushell ...$spans | from json
  → carapace _carapace nushell subcommand receives args
  → complete() in complete.go detects shell "nushell"
  → nushell.Patch(args) strips quotes from args starting with ", ', or `
  → traverse() classifies args (flag, positional, subcommand, dash)
  → storage.get(cmd) resolves the Action
  → action.Invoke(context) resolves callback chain
  → shell.Value("nushell", ...) applies filtering, messages, nospace
  → nushell.ActionRawValues(currentWord, meta, values) formats as JSON
  → JSON output piped through | from json in nushell
  → Nushell displays completion candidates
```

### Step-by-Step

1. **TAB pressed** — nushell's completion system invokes the registered external completer closure
2. **Closure execution** — `carapace $spans.0 nushell ...$spans | from json` runs as a nushell pipeline
3. **Argument construction** — `$spans.0` is the command name; `...$spans` spreads all spans as separate CLI arguments
4. **Shell detection** — The `nushell` argument tells carapace to use the nushell formatter
5. **Patch** — `nushell.Patch()` strips quotes from args that start with `"`, `'`, or `` ` ``
6. **Traversal** — `traverse()` walks the cobra command tree to find the right Action
7. **Invocation** — The Action's callback is resolved to produce RawValues
8. **Formatting** — `shell.Value()` applies filtering, message integration, nospace propagation, then calls `nushell.ActionRawValues()`
9. **JSON output** — `ActionRawValues()` sanitizes, quotes, adds nospace spaces, converts styles, and marshals to JSON
10. **Pipeline** — nushell's `from json` parses the JSON into nushell records
11. **Display** — nushell's completion menu shows the records with value, description, and style
